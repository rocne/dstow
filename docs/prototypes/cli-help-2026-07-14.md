## Top-level help

```
dstow — deploy dotfiles and configuration as symlinks, from packages in repos

Usage:
  dstow <command> [args] [flags]

Deploy:
  stow        Link packages into their targets
  unstow      Remove packages' links from their targets
  restow      Unstow, then stow again (refresh links)
  adopt       Import an existing file into a package, leaving a link behind

Repos:
  repo add     Register a repo from a source (path, URL, github:owner/name)
  repo remove  Unregister a repo (deletes managed clones only)
  fetch        Download remote repo changes; touch nothing on disk
  apply        Fast-forward remote repos to what fetch downloaded

Inspect:
  list        What is configured: repos, packages, targets (never walks disk)
  status      What is deployed: live state of packages against their targets
  deps        Declared dependencies and whether each is present on PATH

Maintain:
  check       Verify every link in the ledger; classify broken and orphaned
  clean       Execute exactly what check reported (broken freely, orphans ask)
  rebuild     Reconstruct a lost ledger by walking configured targets (rare)

Setup:
  snippet     Print a POSIX-sh bootstrap snippet for your shell rc file
  completion  Generate shell completion (bash, zsh, fish, powershell)
  version     Print version

Global flags:
      --color <when>   Colorize output: auto (default), always, never
  -q, --quiet          Suppress informational output (announcements survive)
  -y, --yes            Assume "yes" at confirmation prompts
      --json           Machine-readable output (where a command supports it)
  -h, --help           Help for dstow or any command

Environment:
  DSTOW_PATH   Colon-separated repos prepended to the configured search path

Packages resolve from configured repos; the working directory never changes
what a command does. Run 'dstow <command> --help' for details and examples.
```

## Per-command help

### stow / unstow / restow

```
Link packages into their targets.

Usage:
  dstow stow <package>... [flags]
  dstow stow --all

Each package succeeds or fails on its own; the run continues past failures
and exits nonzero if any package failed. Naming a package explicitly
overrides its exclude-from-bulk setting.

Flags:
      --all          Every package in the resolved set (shadow winners only;
                     excluded repos/packages skipped)
      --repo <repo>  Resolve packages from this repo, bypassing shadowing
  -n, --dry-run      Show the plan; change nothing

Examples:
  dstow stow zsh git tmux
  dstow stow --all
  dstow stow --repo work/dotfiles zsh    # the shadowed one, explicitly
```

(unstow and restow read identically — same args, same flags, mirrored verbs.)

### adopt

```
Import an existing real file into a package; a link takes its place.
Live content always wins — adopt never destroys running configuration.

Usage:
  dstow adopt <file> <package> [flags]
  dstow adopt --occupied <package>       Adopt every occupied path of a package
  dstow adopt --candidates <file>        List packages that could adopt a file

Shows its plan first. Asks before overwriting package content that differs
(--yes accepts, --force also overwrites without a plan confirmation).

Flags:
      --occupied         All occupied paths of the named package
      --candidates       Rank candidate packages for a file; change nothing
  -n, --dry-run          Show the plan; change nothing
      --force            Overwrite differing package content without asking

Examples:
  dstow adopt ~/.zshrc zsh
  dstow adopt --candidates ~/.config/foo/foo.toml
```

### repo add

```
Register a repo — where packages come from.

Usage:
  dstow repo add <source> [flags]

Sources: a local directory path, a full git URL/ssh form, or a qualified
source like github:owner/name (bare owner/name asks first; in scripts,
qualify). Remote sources clone into the managed directory; local paths are
registered in place and never modified.

Adding stows nothing: dstow announces the repo's packages and any shadowing
changes. Re-adding a present repo is a safe, announced no-op.

Flags:
      --stow   After adding, bulk-stow this repo's packages (exclusions apply)

Examples:
  dstow repo add ~/dotfiles
  dstow repo add github:rocne/dotfiles
  dstow repo add rocne/dotfiles --stow
```

### repo remove

```
Unregister a repo. Managed clones are deleted; local-path repos are only
forgotten — your directory is never touched.

Usage:
  dstow repo remove <repo> [flags]

Refuses while the repo still has stowed links (offers unstow-then-remove),
and refuses to delete a managed clone holding work not present at its
source. Each refusal names its remedy.

Flags:
      --unstow   Unstow the repo's packages first, without prompting
      --force    Override both guards (unsaved work will be lost)

Examples:
  dstow repo remove rocne/dotfiles --unstow
```

### fetch / apply

```
Sync remote repos in two explicit phases. Neither ever runs on its own.

Usage:
  dstow fetch [<repo>...]      Download changes; alter nothing on disk
  dstow apply [<repo>...]      Fast-forward clean clones; report old -> new

With no arguments, both act on every remote repo. fetch touches the network
and nothing else — afterwards, status reports behind/ahead. apply is
fast-forward only: divergence or local work refuses loudly, with no stash,
merge, or rebase negotiation. apply never re-stows; structural drift shows
up in status. Linked files change the moment apply moves them — fetch first,
review, then apply.

Examples:
  dstow fetch
  dstow status
  dstow apply rocne/dotfiles
```

### list

```
What is configured — repos, packages, targets, exclusions, sources.
Reads only configuration: instant, side-effect free, no disk inspection.
Deployment truth lives in 'dstow status'.

Usage:
  dstow list [flags]

Flags:
      --repos      Repos only (source, scheme, bulk-exclusion)
      --packages   Packages only (repo-attributed; shadowed duplicates marked)
      --json       Machine-readable listing
```

### status

```
What is deployed — expected links against what targets actually hold.

Usage:
  dstow status [<package>...] [flags]

Package states: stowed, partially stowed, not stowed, occupied, damaged —
plus a drifted marker when the deployed shape differs from what current
config would produce. damaged is only ever claimed with ledger evidence.
Remote repos also show behind/ahead as of the last fetch. Unmet
dependencies appear in package detail.

Flags:
      --repo <repo>   All packages of one repo
      --paths         Per-path detail: each occupant and its remedy
      --json          Machine-readable status
```

### check / clean / rebuild

```
Verify every ledgered link — instant, no tree walk.

Usage:
  dstow check [flags]

Classifies stale links: broken (destination gone) and orphaned (resolves
into a known repo, but no current package config would produce it). clean
executes exactly this report — the two can never disagree.

Flags:
      --json   Machine-readable report
```

```
Execute exactly what check reported.

Usage:
  dstow clean [flags]

Broken links are removed freely. Orphans are shown and confirmed one by one
(--yes removes them without asking). Nothing else is ever touched.
```

```
Reconstruct a lost ledger by scanning configured targets for links into
known repos. The only full tree walk dstow has; explicit and rare.

Usage:
  dstow rebuild
```

### deps

```
Declared dependencies, checked for presence on PATH. Never installs.

Usage:
  dstow deps [<package>] [flags]

Output is machine-consumable line-per-dependency (built for the
loop-and-install hook pattern); human-oriented dependency detail lives in
'dstow status'. Exit 0 when everything in scope is present, 1 when
anything is missing.

Flags:
      --repo <repo>   A repo's effective dependencies
      --global        Global declarations only
      --all           Everything, everywhere
      --json          JSON instead of line format
```

### snippet

```
Print a POSIX-sh snippet that installs dstow if it is absent.

Usage:
  dstow snippet

Compose it yourself: dstow never edits rc files. The snippet's presence
check is local — a shell where dstow exists starts with zero network
traffic and zero output.

Example:
  dstow snippet >> ~/.bashrc
```

## Decision ledger — react to these

Each entry: the call made in the draft above, then alternatives. Numbered
for drive-by reactions ("D1 yes, D3 use the flag, D7 rename").

**D1 — Sync verbs: top-level `fetch` / `apply`.**
REQUIREMENTS §6 already names the phases "fetch" and "apply"; git-lineage
`fetch` is exact (network, touches nothing), and `apply` is honest about
"make it so". Alternatives: `update`/`upgrade` (apt/brew muscle memory, but
`dstow upgrade` reads as upgrading dstow itself — a v2 self-update collision
waiting to happen); nesting as `repo fetch`/`repo apply` (more correct
taxonomy, more typing for a daily-driver pair).

**D2 — Bulk is explicit `--all`; a bare `dstow stow` is a usage error** that
names `--all`. Explicit-over-implicit (§1.6); a typo'd package list should
never silently become "everything".

**D3 — Shadow qualification is a flag: `--repo <repo> <package>`.**
Grammar alternative: `repo:package` or `repo/package` compound args — denser,
but collides visually with `github:owner/name` scheme syntax and with paths;
and the Qualified-source grammar ticket owns that grammar space, so the
surface shouldn't squat on it. Flag is unambiguous and completable.

**D4 — One `list`, no `repo list`.** §7.1 defines list as one scopeable view;
`dstow list --repos` is the repo listing. Alternative: alias `repo list` to
it for muscle memory (costs a second spelling for the same thing — against
single ownership).

**D5 — Candidate helper lives on adopt: `adopt --candidates <file>`.**
It answers "who could adopt this?", so it belongs to adopt's help page.
Alternative: a separate `candidates` command (discoverable alone, but
orphaned from the verb it serves).

**D6 — Bootstrap command is `snippet`.** Literal, matches §10.1's own noun.
Alternatives: `bootstrap` (grander than printing 3 lines), `shellenv`
(brew precedent but a concept mismatch — this isn't env setup; naming
principle forbids established-term-for-wrong-concept).

**D7 — Dependency query is `deps`, machine-first.** CONTEXT.md blesses
"deps" informally; the command's *purpose* is the loop-and-install pattern
(§9.2.4), so line-format machine output is the default and the human view
stays in `status`. Alternatives: `dependencies` (formal, long), human-first
default with `--porcelain` (git precedent, but then the primary consumer
needs a flag).

**D8 — `--dry-run` exists on stow/unstow/restow/adopt only.** The engine is
plan-first so it's nearly free, and "show me" before a first bulk stow is
real value. check/clean already split plan from action, so they don't need
it. Alternative: no dry-run in v1 (leaner; status mostly answers it).

**D9 — Confirmation vs guard: `--yes` is global assume-yes for prompts;
`--force` is per-command and only where REQUIREMENTS names an override**
(repo remove, adopt, clean). `--yes` never bypasses a guard and never
resolves ambiguity — ambiguous input in a non-interactive context stays a
hard error (§1.2). Alternative: no global --yes, only per-command flags
(more explicit, more flags to learn).

**D10 — `clean` needs no `--orphans` flag.** clean executes check's whole
plan; orphan removal is gated by confirmation (or `--yes`), not by opt-in
scope. Alternative: `--orphans` as explicit opt-in with default
broken-only (safer-feeling, but breaks "clean executes exactly check's
plan" — §8.3's one-owner rule).

**D11 — Add-and-stow is `repo add --stow`.** Behaviorally add + bulk stow
scoped to the repo (§5.2.4). Alternative: none serious.

**D12 — Config knobs get no CLI flag twins.** No `--target`, `--no-folding`,
`--dotfiles`, `--ignore` flags — those are config-chain knobs (§4), and a
flag twin would create a second owner for every one of them (and folding is
global-only by §3.3; an invocation flag would quietly reintroduce per-run
contradictions). Stow refugees who type `--no-folding` get a refusal that
names the config knob. This is the draft's most opinionated call.

**D13 — `--json` is a per-command flag** (list, status, check, deps), not
global — `--json stow` has no meaning. Final shape belongs to Output
design; the surface just reserves the spelling.

**D14 — `status --paths`** carries §7.2.4's per-path detail (occupant +
remedy). Alternative: detail-by-default when a single package is named.
