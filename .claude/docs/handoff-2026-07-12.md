## Inputs the design map inherits

1. **REQUIREMENTS.md §11 delegates to design**: CLI syntax and command
   spellings, config file format/schema, metadata dir name/layout,
   qualified-source grammar (`github:owner/name`), ledger format,
   snippet/installer mechanics, output design, Go architecture, release-ci
   wiring (installer script is a v1 *promise*, §10.2).
2. **The gostow audit's verdict**: the engine is operationally sufficient
   (apply ops, per-call fold/dotfiles, additive `Options.Ignore`, plan-first
   atomicity, adopt primitive, `FixQuirks`). Four gaps are design-effort
   inputs — each needs a build-in-dstow vs. gostow-work-item decision:
   - **Pure expected-set planning** (the load-bearing one): `Simulate` yields
     a disk-relative delta, not the expected link set that status/drifted/
     check/adopt-candidates need. Preferred remedy: a gostow plan-only entry
     point; bridge: simulate against an empty scratch target.
   - **Structured conflicts**: occupied path + occupant kind live only in
     prose `Message`s (stable, parity-pinned, but prose).
   - **Ownership introspection**: `findStowedPath` et al. are unexported;
     status attribution and rebuild need a public helper or ~30 risky lines.
   - **Ignore seams** (lesser): built-in list unexported; implicit
     `$HOME/.stow-global-ignore` read can't be disabled per call.
   Note: gostow changes were out of the *requirements* scope, not the design
   effort's. All four gaps are already filed as gostow feature requests:
   [gostow#33](https://github.com/rocne/gostow/issues/33) expected-set
   planning, [#34](https://github.com/rocne/gostow/issues/34) structured
   conflicts, [#35](https://github.com/rocne/gostow/issues/35) ownership
   helper, [#36](https://github.com/rocne/gostow/issues/36) ignore seams.
   The design map decides build-vs-wait per gap.
3. **Standing constraints** (map Notes): dstow owns all config, gostow stays
   config-less; two repos, dstow imports gostow; distribution inherits
   rocne/release-ci (dot-dagger/gostow precedent); design philosophy (single
   ownership, unified I/O, libraries over hand-rolling, conventions).

## Working conventions that persist (also in memory)

- Only ticket resolutions are decisions. **The work wrapper is not evidence**
  — never cite it as authority (user corrected this twice).
- Spelling: `dstow`/`DSTOW` only.
- The user answers by voice — confirm ambiguous load-bearing sentences.
- Prose immediately before an AskUserQuestion dialog is invisible — deliver
  analysis as a plain turn first.
- Concise answers: short paragraphs; the user flagged wall-of-text parsing
  fatigue once.
- Grilling: one question at a time, always with a recommendation; the user
  frequently improves on it (scheme, ledger, dependency naming all came from
  their pushback).
